# BizKnot 現在の開発構成（アーキテクチャ）と選定理由

本ドキュメントでは、現在のBizKnotシステム（モックアップ・MVPフェーズ）における開発構成と、なぜその構成に至ったのかの設計思想を定義します。
実装（ソースやインフラ設定）とこのドキュメントは完全に同期（＝）しています。

## 1. 全体アーキテクチャ概要

現在のBizKnotは、サーバー管理を一切不要とする**完全なサーバーレス・アーキテクチャ**を採用しています。

### 1-1. フロントエンド（UI/UX層）
*   **技術スタック**: HTML5 / Vanilla JavaScript / Tailwind CSS
*   **選定理由**: 
    *   **俊敏性（アジリティ）**: React等のフレームワーク環境構築にかかる時間をゼロにし、即座にモックアップを作成・改修するため。
    *   **Tailwind CSS**: ユーティリティファーストなCSSフレームワークにより、モダンで美しいUI（影、角丸、レスポンシブ）を爆速で構築するため。

### 1-2. バックエンド・インフラ層（BaaS）
*   **技術スタック**: Firebase (Hosting, Authentication, Firestore)
*   **選定理由**: 
    *   **Firebase Hosting**: フロントエンドの静的ファイルを世界中のCDNに高速配信でき、SSLも標準対応。
    *   **Firebase Authentication**: Googleアカウント連携（SSO）を数行のコードで実現でき、セキュアな認証基盤を即座に構築可能。
    *   **Firestore**: リアルタイムなNoSQLデータベースであり、フロントエンドから直接データのCRUD操作が可能。バックエンドAPIの開発工数をまるごと削減。

## 2. 重要なアーキテクチャ上の決定（完全無料枠への対応）

本プロジェクトにおいて最も重要な制約事項は**「Firebaseの無料枠（Sparkプラン）内でシステム全体を稼働させること」**です。
この制約をクリアするため、一般的なアーキテクチャとは異なる以下の特殊な設計を採用しています。

### 2-1. 画像ストレージの廃止とBase64化
*   **課題**: 名刺画像の保存にCloud Storageを利用しようとしたところ、GCPの仕様変更により無料枠での利用が制限（Blazeプランへの課金登録が必須）されました。
*   **解決策**: フロントエンドの `Canvas API` を用いて、撮影した名刺画像をリサイズ（最大1200px）し、JPEG（画質70%）に圧縮。その画像データを **Base64文字列** に変換し、そのままFirestoreのドキュメント内に直接格納するアーキテクチャに変更しました。
*   **効果**: ストレージ料金をゼロにしつつ、CORSエラー等の面倒なインフラ設定も不要になりました。

### 2-2. サーバーレスロジック（Cloud Functions）の廃止
*   **課題**: ユーザー権限（管理者/一般）の付与や、監査ログのセキュアな記録には通常サーバーサイドのロジック（Cloud Functions）が必要ですが、これも無料枠では利用できません。
*   **解決策**: 認証後のアクセス制御（RBAC）および監査ログの書き込み処理をすべてフロントエンド主導で行い、Firestoreの強力な**セキュリティルール（Security Rules）**によって不正な書き込みを防ぐ構成としました。

## 3. その他の特筆すべき実装構成

*   **インメモリ検索**: Firestoreは複数フィールドの曖昧検索（LIKE検索）をサポートしていません。そのため、名刺データ一覧を取得した後に、フロントエンドのメモリ上（JavaScript）でフィルタリングを行うインメモリ検索を実装し、高い検索体験を実現しています。
*   **排他制御（同時編集ブロック）**: 複数人が同じ名刺を編集し、後勝ちでデータが消失するのを防ぐため、保存時に「開いた時点の更新日時」と「DBの最新日時」を比較する楽観的ロックの仕組みを導入しています。
